This is one of the first personal UI design projects I did, and a lot has changed since then. I am happy to say that the app has now been redesigned in line with some of the design changes I have suggested here.

The Challenge

Naturbasen is a citizen science project, where users can report animal sightings on their phone when they are in nature or walking around a city. Users are also able to see where they are, what species have been reported in the area, if the area is a sanctuary or protected, and they can take a picture of the animal and have the community help them identify it for them. I set out to redesign Naturbasen as I wanted to explore how it would be like to work as a digital designer, that is to say, mainly working with the graphical design of an interface.

Criteria for Success

I had some different criteria in mind, from the outset of the project:

  • The process should be fun.
  • I should be able to do the complete redesign in a week.
  • I had to feel proud of the end result.

I also wanted to become more proficient at creating digital prototypes as well as develop my intuition for graphic design.

The Process

I downloaded the app, took screenshots of each screen, and created a prototype of the app in Figma, my design tool of choice. This provided me with an overview of what had to be designed. From what I'm told, a digital designer usually works from a set of wireframes, handed over by the UX designers.

First Impressions

I inspected the app, while imagining the different use cases and writing down my thoughts as I went along. I wrote notes on ideas for the graphical design, but most of my notes were about the usability, and how I thought it could be improved. Whenever I had the opportunity, I showed the app to others, asked them to use it and what they made of it. I found that some of my inital suspicions were confirmed - there was room for improvement on the usability side of things.

Working in Figma

I spent the remaining time working in Figma, creating the different screens in a more or less sequential process. I broke the design down into the different components which were used multiple times throughout the design. This made it easy to update the entire design across all screens, when making an adjustment to a main component, say the main component of the header. But it also allowed for an easy way to make alternative designs - components could easily be swapped out. Comparing different versions in order to figure out what worked and what didn't was the underlying process of creating the design.

Creating a Design System

This project was the first time I came across the notion of a design system. A design system, to me, is a document showing what the diffent UI elements look like, in order to reach a consistency throughout the product. It is a shared reference point for designers and developers, it helps designers share and reuse designs and it enables developers to inspect UI components. Eventhough this was an individual project, I find that it is good practise and that it does help with providing an overview. I used Atomic Design by Brad Frost, and Google's Material Design as inspiration for creating my design system.

Design Choices

Colours

I've chosen to still go with blue, since it seems to be a big part of the visual identify of the product, but I did consider going green. I've tried to move away from the different shades of blue, and instead opt for white in many places, which I find helps a lot with the contrast, and a more clear and simple visual design.

Where am I?

Knowing your location seems pivotal for the app, seeing as a map is accesible on multiple screens. I decided to move the map to the landing page, so it is accesible from a single place, removing the "where am I?" screen. Having the map on the landing page also gives some much needed context to the icons. These icons light up according to the type of area the user is currently located in. For example, if the user is currently standing in a bird sanctuary, the corresponding icon will light up.

Search

To be consistent with other apps, and to free up space, I decided to move the search function to a search bar, placed at the top of the screen. This consistency, together with the fact that it is visually distinct from the buttons, hopefully makes it easier to find for users.

What can I see here?

I added a picture for each list item, giving the user some visual feedback, along with making the app more visually appealing. A common use case might be that the user wants to know what species they can see in the area, not knowing the names. Having a picture helps out. They might also use the function when spotting an animal, wanting to narrow down the possibilities. Here it also makes sense to have a picture, making it easier for the user to report the animal, and making them more confident that they picked the right one. I also added a subtitle with the taxonomic name of the species.

Protected areas

If you are in a protected area, the respective icon will light up. Since it is possible that the user is in an area which is protected in more that one way, the user can select which area should be lit up on the map. This is done by pressing an active icon, which prompts a menu with the given active areas. Here the user can see more information about what type of area they are located in, and they can switch between which type is shown on the map. If the user presses an icon which is not lit up, a messsage is show about the type of area the icon indicates. This was an important change from the original design, since nothing happened when trying to press an icon.

Report

Lists with many elements should be ordered in some way, which makes it easier and faster for a user to sort through. So, I ordered the list of species and the input lists alphabetically. Secondly, I added an image of the species which the user is about to report. This provides the user with some extra feedback, letting them know that they did indeed select the species they had in mind. When reporting, I added an 'edit' function, having the following use case in mind: the user is reporting a species, but discovers that they have put the wrong name, or location. They want to easily change this, without going back and discarding the other information, which they have already spent time providing. These last two changes conform to Jakob Nielsen's 10 usability heuristics, of helping users recognize, diagnose, and recover from errors.

My observations

The update funciton is system-oriented instead of user-centered - when pressing "update", a lengthy message appears explaining what the system is doing, with a warning about using the function. Instead, the app should automatically try to synchronize with the online database, whenever there is internet coverage. In order to provide feedback about this, a system message should be displayed in the bottom, notifying the user if there is coverage, it is trying to synchronise, or if it completed synchronising.

Possible Improvements

Ideally, user testing should be made after any changes to the design, including changes to the graphical design, as it has an affect on the user experience. However, since my personal goal for this case was to develop an intution for graphic design and become more proficient at prototyping, I chose to not do any follow up testing with users. Because of this, I have reflected on some of the bigger changes I have made to the app.

It could be that having the map on the landing page is too much, as it already has a lot of information on it. Having the map on the landing page, might also affect the user experience in a different way - it might drain the phone battery more quickly, as more resources are spent showing the map.

The protected area icons might also be improved. After having showed my design to several people, I realise that even though the area types are better explained, the colored areas on the map tend to be overlooked. This might be a case of too much information on the landing page, or it could be that the colours are not contrasting enough. On the other hand, maybe it is not so critical that the user does not spot it at the first glance.

Final Reflections

Creating alternatives and comparing them is a great way to know that you are making the right decision, but developing an intuition about graphic design can make the process more effecient. I find that it is something that comes with practise, and why projects like these have been important for my skills to get where they are today.

Overall, I find the design more visually appealing than the existing one. The design could have been more bold, opting for green instead of blue for example, and more stylised fonts. However, considering the demographic of users, I think I made the right choices.

The project took about 7 workdays, but I have made some adjustments following that. I enjoyed working on it very much, and found that I often lost track of time and ran out of music to listen to, while working on it. That might have been the most important outcome, for me personally.

I quickly found that I focused a lot on the user experience and my knowledge of interaction design informed some design changes not related to the graphical design. In fact, I think that the value of the final product is at least as much in the usability changes as in the updated visual design.

In conclusion, I developed my skills with graphic design, but the main outcome of the project is that I should focus more on what I'm good at. I hope that you enjoyed the read, and if you haven't already, check out the prototype.
