The Challenge

Naturbasen is a citizen science project, where users can report animal sightings on their phone, when they are in nature or walking around a city. Users are also able to see where they are, what animals have been reported in the area, if the area is a sanctuary or protected, and they can take a picture of the animal and have their smartphone try to identify it for them. I set out to redesign Naturbasen as I wanted to explore how it would be like to work as a digital designer, that is to say, mainly working with the graphical design of an interface.

Criteria for Success

I had some different criteria in mind, from the outset of the project:

  • The process should be fun.
  • I should be able to do the complete redesign in a week.
  • I had to feel proud of the end result.

I also wanted to become more proficient at creating digital prototypes as well as develop my intuition for graphical design.

The Process

I downloaded the app, took screen shots of each screen, and created a prototype of the app in Figma, my design tool of choice. This way I had an overview. From what I'm told, a digital designer usually works from a set of wireframes, and this was kind of like that - working from the standpoint of the information architecture of the app.

First Impressions

I inspected the app, while imagining the different use cases and writing down my thoughts as I went along. I wrote notes on ideas for the graphical design, but most of my notes were about the usability, and how I thought it could be improved. Whenever I had the opportunity, I showed the app to others, asked them to use it and what they made of it. I found that some of my inital suspicions were confirmed - there was room for improvement on the usability side of things.

Working in Figma

I spend the remaining time working in Figma, creating the different screen in a more or less sequential process. I broke the design down into the different components which were used multiple times throughout the design. This made it easy to update the entire design across all screens, when making an adjustment to a main component, say the main component of the header. But it also allowed for an easy way to make alternative designs - components could easily be swapped out. Creating different designs and comparing them in order to figure out what worked and what didn't was the underlying process of creating the design.

Creating a Design System

This project was the first time I came across the notion of a design system. A design system, to me, is a document detailing what the diffent UI elements look like, so it is consistent throughout the product. It is a shared reference point for designers and developers, it helps designers share and reuse designs and developers inspect the UI components. Eventhough this was an individual project, I created a design system, for good practice if nothing else. I did find that it helps create an overview. I used Atomic Design by Brad Frost, and Google's Material Design as inspiration for creating my design system.

Design Choices

Where am I?

Knowing your location seemed pivotal for the app, seeing as a map is to be found on several of the screens. I decided to move the map to the landing page. This decision was also guided by the proximity principle, as it gives some much needed context to the icons. These icons light up according to the type of area the user is currently located in, e.g. a bird sanctuary.

Search

To be consistent with other apps, and to free up space, I decided to move the search function to a search bar, placed in the top of the screen. This consistency, together with the fact that it is visually distinct from the buttons, hopefully makes it easier to find, for users.

What can I see here?

I added a picture for each list item, giving the user some visual feedback, along with making the app more visually appealing. A common use case might be that the user wants to know what animals they can see in the area, not knowing the names. Having a picture helps out. They might also use the function when spotting an animal, wanting to narrow down the possibilities. Here it also makes sense to have a picture, making it easier for the user to report the animal, and making them more confident that they picked the right one. I also added a subtitle with the taxonomic name of the animal.

If you are in a protected area, the repective icon will light up. Since it is possible that the user is in an area which is protected in more that one way, the user can select which area should be lit up on the map. This is done by pressing an active icon, which prompts a menu with the given active areas. Here the user can see more information about what type of area they are located in, and they can switch between which type is shown on the map. If the user presses an icon which is not lit up, a messsage is show about the type of area the icon indicates. This was an important change from the original design, since nothing happened when trying to press an icon.

Report

  • Lists with many elements should be sorted alphabetically, so it faster to search through.
  • Added a default image of the animal which the user is about to report. It provides some extra feedback to the user, ensuring them that they have selected the right animal.
  • Added an 'edit report' funktion. Use case: user is reporting an animal, but discover that they have put in the wrong species or location. They should be able to easiely go back and change this, without discarding the other information about the report. In other words, it allows the user an easy way to correct mistakes.

My observations

The update funciton is system-oriented instead of user-centered - when pressing "update", a lengthy message appears explaining what the system is doing, with a warning about using the function. Instead, the app should automatically try to synchronize with the online database, whenever there is internet coverage. In order to provide feedback about this, a system message should be displayed in the bottom, notifying the user if there is coverage, it is trying to synchronise, or if it completed synchronising.

Possible Improvements

I did't conduct any user testing following the biggest change I made to the app - the landing page. This should ideally be done, but since the focus was mainly (at least in the beginning) on the graphical design, I let it slide.

It could very well be that I put too much information on the landing page, and that it makes more sense to have a seperate page for the map. Having the map on the landing page, might also affect the user experience in a different way - it might use up more power, if the user spends more time on the landing page, and displaying the map is costly, computational wise.

After having showed my design to several people, I realise that even though the area types are better explained, the colored areas on the map tend to be overlooked.

Final Reflections

Creating alternatives and comparing them is a great way to know that you are making the right decision, but developing an intuition about graphical design can make the process more effecient. I find that it is something that comes with practise, and why projects like this one has been important for my skills to get where they are today. Jeg syntes at resultatet var en klar forbedring af det forhenværende design, især med de nye farver, som virker mindre som udvasket blå og i stedet er gået hen og blevet mere kontrastfyldt. Tidsmæssigt tog det mere end en uge fra starten af projektet til slutningen. Det var mere omkring to. Men siden det ikke var alle dage jeg arbejdede 8 timer på projektet, føler jeg at den endelige arbejdsindsats lå på omkring 7 arbejdsdage, hvilket jeg så som en succes. Nok det vigtigste kriterie som jeg havde sat mig selv, blev opfyldt - nemlig at det skulle være sjovt at arbejde med. Der var dage hvor jeg fuldstændigt mistede tidsfornemmelsen og løb tør for musik album at lytte til, imens jeg arbejdede på projektet. Og det var nok det største udbytte.

I forhold til spørgsmålet om specialisering vs generalisering, så blev spørgsmålet svaret på rimelig hurtigt undervejs. Værdien i det endelige produkt ligger for mig mindst lige så meget i de forbedringer i brugeroplevelsen som i det grafiske redesign. Det er en lektion som senere hen er gået igen til de kaffemøder jeg har været til. Man skal fokusere på det man er god til. Hvis man er mere generalist end specialist, giver det mere mening at søge ind steder hvor der er brug for en bredere faglighed, end at man konkurrere med andre som f.eks. har en specifik baggrund inden for f.eks. grafisk design. Grafisk design er stadig noget jeg finder meget spændende, og noget som jeg både vil lære mere om og blive bedre til, men i mit tilfælde giver det mening også at ligge vægt på mine kompetencer inden for f.eks. brugerinddragelse.
